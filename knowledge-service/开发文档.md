## 1. 模块概述

### 1.1 定位

`knowledge-service` 是"基于大模型的个性化资源生成与学习多智能体系统"中负责**知识库管理**的独立模块。它从 Next.js 主应用中抽离为独立的 Python 微服务，通过 REST API 对外提供语义检索与事实验证能力。

### 1.2 与主应用的关系

```
┌──────────────────────┐         HTTP          ┌──────────────────────┐
│   Next.js 主应用         │ ←──────────────────→ │   knowledge-service   │
│                        │                      │                       │
│  lib/knowledge/       │  POST /api/v1/search  │  Chroma + BGE-M3      │
│    retriever.ts ──────┼──────────────────────→│  语义向量检索          │
│    fact-check.ts ─────┼──────────────────────→│  事实交叉验证          │
│                        │                      │                       │
│  服务不可用时           │                      │  knowledge_base/      │
│  自动降级到本地         │                      │  种子课程文档          │
│  关键词检索             │                      │                       │
└──────────────────────┘                      └──────────────────────┘
```

### 1.3 赛题要求对照

| 赛题要求 | 本模块实现 |
|----------|-----------|
| Chroma 向量数据库 | ✅ 持久化存储，cosine 相似度检索 |
| BGE-M3 中英双语 Embedding | ✅ 1024维向量，懒加载单例 |
| PDF/PPTX 文档解析 | ✅ PyMuPDF + Unstructured + Markdown |
| 至少一门完整高校课程 | ✅ 8篇"数据结构与算法"种子文档 |
| 防幻觉四层机制 | ✅ 第1层(检索约束) + 第3层(事实核查) |

## 4. 快速开始

### 4.1 环境要求

- Python 3.11+
- 至少 4GB 可用内存（BGE-M3 模型首次下载约 2.2GB）

### 4.2 本地启动

```bash
# 1. 创建虚拟环境
python -m venv venv
source venv/bin/activate      # Linux/Mac
# venv\Scripts\activate       # Windows

# 2. 安装依赖
pip install -r requirements.txt

# 3. 导入种子数据（首次）
python seed.py

# 4. 启动服务
uvicorn app.main:app --host 0.0.0.0 --port 8000 --reload
```

### 4.3 验证服务可用

启动后用 `curl` 确认所有接口正常：

```bash
# 健康检查
curl http://localhost:8000/api/v1/health
# → {"status":"healthy","vector_count":487,"embedding_model":"BAAI/bge-m3"}

# 语义检索
curl -X POST http://localhost:8000/api/v1/search \
  -H "Content-Type: application/json" \
  -d '{"query": "数组的时间复杂度", "top_k": 3}'
# → {"results":[{"id":"01-数组-0-0","text":"...","source":"01-数组","score":0.87},...]}

# 事实验证
curl -X POST http://localhost:8000/api/v1/fact-check \
  -H "Content-Type: application/json" \
  -d '{"content": "数组访问时间复杂度为 O(1)，插入为 O(n)", "topic": "数组"}'
# → {"score":100,"flagged":[],"checked":2}

# 知识库统计
curl http://localhost:8000/api/v1/stats
# → {"total_chunks":487,"sources":["01-数组",...],"last_ingested":null}
```

> 4 个接口都返回正常结果，说明服务已就绪。也可浏览器访问 `http://localhost:8000/docs` 查看 Swagger 交互式文档。

### 4.4 添加新课程知识

```bash
# 方法1：放入文件后重索引
cp 新课程.pdf knowledge_base/
python seed.py --clear

# 方法2：通过 API 上传
curl -X POST http://localhost:8000/api/v1/ingest \
  -F "files=@新课程.pdf" \
  -F "files=@补充材料.pptx"

# 方法3：代码调用
```

## 5. API 接口文档

所有接口前缀：`/api/v1`

### 5.1 语义检索 `POST /api/v1/search`

对知识库做语义相似度检索，返回最相关的文档片段。

**请求**：
```json
{
  "query": "数组的时间复杂度",
  "top_k": 5
}
```

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| query | string | 是 | 检索查询文本，1~2000 字符 |
| top_k | int | 否 | 返回数量，默认 5，范围 1~50 |

**响应**：
```json
{
  "results": [
    {
      "id": "01-数组-0-0",
      "text": "## 时间复杂度\n| 操作 | 复杂度 |\n| 按下标访问 | O(1) |...",
      "source": "01-数组",
      "score": 0.8723
    }
  ]
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| results[].id | string | Chunk 唯一标识（格式：`来源-段落-子块`） |
| results[].text | string | Chunk 文本内容 |
| results[].source | string | 来源文档名（不含扩展名） |
| results[].score | float | 余弦相似度 0~1 |

```bash
# 试用
curl -X POST http://localhost:8000/api/v1/search \
  -H "Content-Type: application/json" \
  -d '{"query": "链表插入删除的时间复杂度", "top_k": 3}'
```

### 5.2 事实验证 `POST /api/v1/fact-check`

对 LLM 生成内容做交叉验证，提取"可核声明"回查知识库。

**请求**：
```json
{
  "content": "# 数组\n\n时间复杂度：访问 O(1)，插入 O(n)...\n...",
  "topic": "数组"
}
```

| 参数 | 类型 | 必填 | 说明 |
|------|------|------|------|
| content | string | 是 | 待核查的 LLM 生成内容（Markdown） |
| topic | string | 否 | 主题词，辅助定向检索 |

**响应**：
```json
{
  "score": 80,
  "flagged": ["动态数组扩容摊还复杂度为 O(n log n)"],
  "checked": 5
}
```

| 字段 | 类型 | 说明 |
|------|------|------|
| score | int | 可信度 0~100（有知识库佐证的声明占比） |
| flagged | string[] | 未找到佐证的声明摘要 |
| checked | int | 可核声明总数 |

核查仅关注"可核声明"——包含复杂度记号 `O(...)` 或数值事实的句子。每条声明回查 Chroma，token 重叠度 ≥2 视为有佐证。

```bash
# 试用
curl -X POST http://localhost:8000/api/v1/fact-check \
  -H "Content-Type: application/json" \
  -d '{"content": "数组访问时间复杂度为 O(1)，插入为 O(n)，删除为 O(log n)", "topic": "数组"}'
```

### 5.3 文档导入 `POST /api/v1/ingest`

上传文档或重索引知识库目录。支持两种模式：

**模式1 — 上传文件**：
```
POST /api/v1/ingest
Content-Type: multipart/form-data

files: [file1.pdf, file2.pptx, file3.md]
clear_first: false
```

**模式2 — 目录索引**（未上传文件时）：自动扫描 `KNOWLEDGE_BASE_DIR` 导入。

**响应**：
```json
{
  "ingested": 150,
  "sources": ["01-数组", "02-链表"],
  "errors": []
}
```

支持格式：`.md` / `.pdf` / `.pptx` / `.ppt`

```bash
# 试用
curl -X POST http://localhost:8000/api/v1/ingest \
  -F "files=@新课程.pdf"

# 重索引整个知识库目录
curl -X POST http://localhost:8000/api/v1/ingest

# 清空后重索引
curl -X POST http://localhost:8000/api/v1/ingest \
  -F "clear_first=true"
```

## 6. 数据流与集成方式

### 6.1 检索数据流

```
Agent 调用
  │
  ▼
searchKnowledge(query, k)          [retriever.ts — 异步]
  │
  ├─→ 尝试 POST /api/v1/search    [Python 服务 — Chroma 语义检索]
  │     │
  │     └─→ BGE-M3 向量化 query
  │         → Chroma.query()
  │         → 返回 { results: [...] }
  │
  └─→ 失败/不可用 → 本地回退       [Node.js — 关键词重叠检索]
        │
        └─→ fs.readdirSync("knowledge_base/")
            → CJK 分词 + TF 归一化
            → 返回 KnowledgeChunk[]
```

### 6.2 事实核查数据流

```
runResourceAgent() 生成完毕
  │
  ▼
factCheck(content, topic)           [fact-check.ts — 异步]
  │
  ├─→ 尝试 POST /api/v1/fact-check [Python 服务]
  │     │
  │     └─→ 提取可核声明
  │         → 每条声明回查 Chroma
  │         → 返回 { score, flagged, checked }
  │
  └─→ 失败/不可用 → 本地回退
  │
  ▼
嵌入 GeneratedResource.fact_check
  → SSE "resource" 事件 → 前端 ResourceCard 展示
```

### 6.3 TS 适配层

主应用中与知识库相关的代码集中在两个文件，内部依赖 `knowledge-client.ts` 提供的 HTTP 工具：

| 文件 | 导出函数 | 调用方 |
|------|----------|--------|
| `lib/knowledge/retriever.ts` | `searchKnowledge()`(async), `formatContext()`, `resetKnowledgeCache()` | resource-runner, planner-agent, tutor-agent |
| `lib/knowledge/fact-check.ts` | `factCheck()`(async) | resource-runner |

**设计原则**：Agent 只需在调用处加 `await`，无需改动业务逻辑。服务不可用时自动降级到本地关键词检索，系统在 mock 模式下仍可运行。

## 7. 接入主应用

> 本章面向主应用（Next.js 项目）的开发者。将知识库服务接入主应用，需要改动 6 个文件、约 30 行代码，5 分钟可完成。

### 7.1 新增 knowledge-client.ts

在主应用的 `lib/knowledge/` 目录下新建 `knowledge-client.ts`：

```typescript
const DEFAULT_BASE_URL = "http://localhost:8000";

export function getServiceUrl(): string {
  return process.env.KNOWLEDGE_SERVICE_URL || DEFAULT_BASE_URL;
}

let _available: boolean | null = null;

export async function isServiceAvailable(): Promise<boolean> {
  if (_available !== null) return _available;
  try {
    const res = await fetch(`${getServiceUrl()}/api/v1/health`, {
      signal: AbortSignal.timeout(3000),
    });
    _available = res.ok;
  } catch {
    _available = false;
  }
  return _available;
}

export function resetServiceAvailability(): void {
  _available = null;
}

export async function knowledgeFetch<T>(
  path: string,
  body: Record<string, unknown>,
  timeout = 15000,
): Promise<T> {
  const res = await fetch(`${getServiceUrl()}${path}`, {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify(body),
    signal: AbortSignal.timeout(timeout),
  });
  if (!res.ok) {
    throw new Error(`Knowledge service error ${res.status}`);
  }
  return res.json() as T;
}
```

### 7.2 修改 retriever.ts

将 `searchKnowledge` 改为异步，优先调 Python 服务，失败降级本地：

```typescript
// 文件顶部新增 import
import { isServiceAvailable, knowledgeFetch } from "@/lib/knowledge/knowledge-client";

// KnowledgeChunk 接口新增可选字段
export interface KnowledgeChunk {
  id: string;
  text: string;
  source: string;
  score?: number;  // 新增
}

// searchKnowledge 改为 async
export async function searchKnowledge(query: string, k = 5): Promise<KnowledgeChunk[]> {
  try {
    const available = await isServiceAvailable();
    if (available) {
      const data = await knowledgeFetch<{
        results: { id: string; text: string; source: string; score: number }[];
      }>("/api/v1/search", { query, top_k: k });
      if (data?.results?.length) {
        return data.results.map((r) => ({
          id: r.id, text: r.text, source: r.source, score: r.score,
        }));
      }
    }
  } catch { /* 降级 */ }
  return localSearch(query, k);  // 原同步逻辑改名
}
```

### 7.3 修改 fact-check.ts

将 `factCheck` 改为异步：

```typescript
import { isServiceAvailable, knowledgeFetch } from "@/lib/knowledge/knowledge-client";

export async function factCheck(content: string, topic: string): Promise<FactCheckResult> {
  try {
    const available = await isServiceAvailable();
    if (available) {
      const data = await knowledgeFetch<{
        score: number; flagged: string[]; checked: number;
      }>("/api/v1/fact-check", { content, topic });
      return { score: data.score, flagged: data.flagged, checked: data.checked };
    }
  } catch { /* 降级 */ }
  return localFactCheck(content, topic);  // 原同步逻辑
}
```

## 8. 防幻觉机制

本模块直接承担赛题四层防幻觉机制中的**第1层**和**第3层**：

| 层级 | 机制 | 实现位置 | 说明 |
|------|------|----------|------|
| 第1层 | 检索约束 | `services/retriever.py` | 所有 LLM 调用前先检索知识库，结果作为 Prompt 上下文 |
| 第2层 | Prompt 约束 | 主应用 `resource-runner.ts` | `ANTI_HALLUCINATION_RULES` 强制"仅基于知识库"（不在本模块） |
| 第3层 | 事实核查 | `services/verifier.py` | 生成后提取声明 → 回查 Chroma 交叉验证 → score + flagged |
| 第4层 | 引用标注 | 主应用 `resource-runner.ts` | `sources` 字段标注来源（不在本模块） |

第3层事实核查的核心逻辑：

1. 清洗生成内容（去代码块、标题、引用标记）
2. 按句切分，筛选"可核声明"（含 `O(...)` 复杂度或数值的句子）
3. 每条声明回查知识库（`search(claim, top_k=2)`）
4. token 重叠度 ≥2 → 有佐证；否则 → 标记为 flagged
5. 计算 score = 有佐证数 / 可核总数 × 100

## 11. 常见问题

**Q: 首次启动很慢？**  
A: BGE-M3 模型首次下载约 2.2GB，需等待几分钟。Docker 构建时已预下载，后续启动快。开发模式用 `--reload` 保持模型常驻内存。

**Q: 知识库服务没启动，主应用还能用吗？**  
A: 能。主应用会自动降级到本地关键词检索（读取 `knowledge_base/*.md`），功能不受影响，只是检索精度降低。

**Q: 如何更换 Embedding 模型？**  
A: 修改环境变量 `EMBEDDING_MODEL_NAME`（如改为 `BAAI/bge-large-zh-v1.5` 纯中文模型）。注意需要清空 Chroma 重建索引（`python seed.py --clear`），因为不同模型的向量维度不同。

**Q: 如何支持更多文档格式？**  
A: 在 `services/ingester.py` 的 `_read_file()` 函数中增加新的扩展名分支，添加对应的解析器即可。
