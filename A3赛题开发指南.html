01 赛题深度解读与核心要求拆解

1.1 赛题本质

这道题的核心是构建一个多智能体协作的个性化学习平台，利用大模型（LLM）为高校学生自动生成多模态学习资源，并根据学生画像提供个性化学习路径。出题企业是科大讯飞，因此在技术选型上需要重点考虑讯飞生态工具。

⚠️ 赛题关键约束

赛题原文明确要求："开发过程中使用的其他AI辅助工具，需选用科大讯飞相关工具。" 这意味着你的AI编程助手、大模型API调用应优先使用讯飞产品生态（如 iFlyCode、星火大模型 API、讯飞开放平台各项能力等），这在评审时会是加分点。

1.2 五大核心功能拆解

编号 功能模块 性质 核心要求
1 对话式学习画像构建 必做 通过自然语言对话（非表单）自动抽取学生特征，构建≥6个维度的动态画像，支持随学随更新
2 多智能体协同资源生成 必做 不同角色智能体协作生成≥5种类型资源（讲解文档、思维导图、题库、拓展阅读、视频/动画、代码实操等）
3 学习路径规划与推送 必做 多智能体协同整合资源，动态规划学习路径，基于画像精准推送多类型内容
4 智能辅导 加分项 即时多模态答疑（文字+图解+短视频），针对性学习引导
5 学习效果评估 加分项 实时跟踪学习行为，多维度评估，动态调整推送策略和学习计划

1.3 非功能性要求

  • 界面规范：流式输出、Markdown渲染、多模态内容卡片化展示，符合现代AI产品交互规范
  • 开源标注：使用开源项目需在文档显著位置标注名称、来源及协议
  • 防幻觉：完善的防幻觉与内容安全过滤机制，确保学术内容无事实性错误
  • 响应速度：核心功能响应时间合理，多模态生成提供"生成进度追踪"或"流式呈现"

1.4 评分标准与权重

评分项 占比 得分要点
创新价值与实用性 35% 选题切入点新颖、解决真实痛点、多智能体协作设计有创新、用户体验有亮点
功能实现及技术要求 45% 5大功能完整实现、多智能体框架明确、AI技术融合深度、系统稳定可运行
配套文档丰富度 10% 系统开发说明书、测试说明书规范完整，含架构图/流程图，需求与技术结合点清晰
演示视频、PPT效果 10% 演示视频≤7分钟、PPT逻辑清晰重点突出，展示多模态生成效果与AI技术融合
💡 策略提示

功能实现占45%是最大权重，必须保证5个核心功能完整可运行。创新价值占35%，建议在多智能体协作机制和多模态资源生成质量上做出亮点。两个加分项（智能辅导+效果评估）能显著拉开差距，建议尽量实现。

📋 关于多智能体框架的选型说明

赛题要求"须明确系统中多智能体协同框架"。推荐 CrewAI 的核心理由：

  • 角色驱动：天然支持为每个Agent定义角色（role）、目标（goal）、背景故事（backstory），完美匹配赛题"不同角色的智能体协作"的要求
  • 任务编排：支持顺序执行（sequential）和层级执行（hierarchical），可用于"路径规划Agent分发任务给资源生成Agent"
  • 工具集成：每个Agent可绑定不同工具（搜索、代码执行、文档生成等），实现5种以上资源类型的分别生成
  • 学习成本低：API设计简洁，文档完善，社区活跃

备选方案：AutoGen（微软，对话式多智能体）或 LangGraph（基于状态图，更灵活但更复杂）。

05 第二步：多智能体框架搭建

这是整个系统的核心骨架。使用 CrewAI 定义所有智能体角色，并建立协作编排机制。

5.1 大模型适配层

首先将讯飞星火大模型适配为 CrewAI / LangChain 可用的 LLM 接口：

# backend/app/core/llm.py
from sparkai.llm.llm import ChatSparkLLM
from langchain_core.language_models import BaseChatModel
from langchain_core.messages import AIMessage, BaseMessage
from langchain_core.outputs import ChatGeneration, ChatResult
from typing import List, Optional, Any
import os

class SparkLLMAdapter(BaseChatModel):
    """将讯飞星火大模型适配为LangChain接口"""

    spark: ChatSparkLLM = None

    def __init__(self, **kwargs):
        super().__init__(**kwargs)
        self.spark = ChatSparkLLM(
            app_id=os.getenv("SPARK_APPID"),
            api_key=os.getenv("SPARK_API_KEY"),
            api_secret=os.getenv("SPARK_API_SECRET"),
            domain=os.getenv("SPARK_DOMAIN", "4.0Ultra"),
            temperature=0.7,
            streaming=True,  # 启用流式输出
        )

    def _generate(self, messages: List[BaseMessage],
                  stop: Optional[List[str]] = None,
                  **kwargs) -> ChatResult:
        # 转换消息格式并调用星火API
        spark_messages = [{"role": m.type, "content": m.content}
                         for m in messages]
        response = self.spark.generate([spark_messages])
        # ...解析响应并返回ChatResult
        pass

    async def _astream(self, messages, **kwargs):
        """异步流式输出，用于SSE前端推送"""
        async for chunk in self.spark.astream(messages):
            yield chunk

    @property
    def _llm_type(self) -> str:
        return "spark-llm"

# backend/app/core/rag.py
from langchain_community.document_loaders import (
    PyMuPDFLoader, UnstructuredPowerPointLoader,
    UnstructuredWordDocumentLoader, WebBaseLoader
)
from langchain.text_splitter import RecursiveCharacterTextSplitter
from langchain_community.embeddings import HuggingFaceBgeEmbeddings
from langchain_community.vectorstores import Chroma
from langchain.tools import Tool

class RAGRetriever:
    """RAG检索引擎：文档加载 → 分块 → 向量化 → 检索"""

    def __init__(self):
        # 使用BGE-M3中文embedding模型
        self.embeddings = HuggingFaceBgeEmbeddings(
            model_name="BAAI/bge-m3",
            model_kwargs={"device": "cpu"},  # 无GPU时用CPU
        )
        self.text_splitter = RecursiveCharacterTextSplitter(
            chunk_size=500,
            chunk_overlap=50,
            separators=["\n\n", "\n", "。", ".", " "]
        )
        self.vectorstore = None
        self.search_tool = None

    def build_index(self, docs_dir: str):
        """从目录加载所有文档并构建向量索引"""
        import glob
        documents = []
        for pdf_path in glob.glob(f"{docs_dir}/*.pdf"):
            documents.extend(PyMuPDFLoader(pdf_path).load())
        for pptx_path in glob.glob(f"{docs_dir}/*.pptx"):
            documents.extend(UnstructuredPowerPointLoader(pptx_path).load())
        for docx_path in glob.glob(f"{docs_dir}/*.docx"):
            documents.extend(UnstructuredWordDocumentLoader(docx_path).load())

        # 分块
        chunks = self.text_splitter.split_documents(documents)

        # 构建向量库
        self.vectorstore = Chroma.from_documents(
            chunks, self.embeddings,
            persist_directory="./chroma_db"
        )

        # 创建检索工具供Agent使用
        self.search_tool = Tool(
            name="knowledge_search",
            description="搜索课程知识库获取准确知识点信息",
            func=self._search
        )

    def _search(self, query: str, k: int = 5) -> str:
        """检索知识库"""
        results = self.vectorstore.similarity_search(query, k=k)
        return "\n\n".join([doc.page_content for doc in results])
💡 RAG是防幻觉的关键

所有资源生成Agent在调用大模型时，都应先通过RAG检索知识库获取准确内容，再将检索结果作为上下文传给大模型。这样生成的文档、题库等内容就有可靠的知识来源，大幅降低幻觉风险。在文档中要重点说明这一机制。

7.1 实现思路

9.2 动态调整机制

学习路径不是静态的。当学生完成某个步骤后（如做完题库），评估Agent分析答题正确率：

  • 正确率 < 60%：回退一步，生成更基础的讲解资源

12 第九步：防幻觉与内容安全机制

赛题明确要求"完善的防幻觉与内容安全过滤机制"，这是非功能性需求中的硬性要求。

12.1 防幻觉机制（四层防护）

层级 机制 技术实现
第一层：检索约束 RAG提供知识来源 所有生成任务先检索知识库，将检索结果作为上下文约束LLM
第二层：Prompt约束 系统提示词限制 在System Prompt中明确"仅基于提供的知识库内容回答，不确定时说明"
第三层：事实核查 生成后交叉验证 生成内容后再用RAG检索相关知识点，检查是否一致
第四层：引用标注 标注知识来源 在生成的文档中标注内容来源（教材页码/章节），便于核查

12.2 内容安全过滤

# 防幻觉Prompt模板
SYSTEM_PROMPT = """你是一个教育内容生成助手。请严格遵循以下规则：

1. 仅基于提供的【知识库内容】生成学习资源，不得编造知识库中不存在的信息。
2. 如果知识库中没有足够信息，请明确说明"当前知识库中暂无此内容"。
3. 生成的内容必须标注知识来源（如"参考：教材第X章"）。
4. 对于数值、公式、代码等关键内容，必须确保与知识库一致。
5. 不得生成任何与学习无关的、敏感的或不当的内容。

【知识库内容】：
{retrieved_context}
"""

# 内容安全过滤
async def content_safety_check(text: str) -> bool:
    """检查生成内容是否安全"""
    # 1. 敏感词过滤
    if contains_sensitive_words(text):
        return False
    # 2. 调用讯飞内容审核API
    result = await spark_content_moderation(text)
    return result["pass"]

17.2 关键避坑指南

⚠️ 常见失分点
  1. 不是"多智能体"：只是简单调用多次大模型API，没有真正的Agent角色分工和协作编排 → 必须用CrewAI等框架明确角色
  2. 资源类型不足5种：只生成了文档和题目 → 至少完成5种，建议6种
  3. 没有RAG：大模型直接生成，内容全是幻觉 → 必须有知识库支撑
  4. 画像维度不足6个：只记录了专业和年级 → 补齐6个维度
  5. 没用讯飞工具：赛题明确要求 → 全链路使用讯飞产品
  6. 系统无法运行：依赖配置复杂 → Docker一键部署
  7. 文档缺图表：全是文字 → 至少5张以上架构/流程图
  8. 没有流式输出：白屏等待 → 所有AI回复都用SSE流式
🏆 最终建议

这道赛题的核心竞争力在于"多智能体协作的真实性"和"多模态生成的丰富度"。多数参赛队伍能做到"调用大模型生成文档"，但能在"多Agent角色分工协作"和"视频/动画等多模态生成"上做出亮点的队伍不多。把精力集中在这两个差异化点上，配合完善的防幻觉机制和讯飞工具链的深度使用，就能在45%+35%=80%的核心评分中取得优势。
